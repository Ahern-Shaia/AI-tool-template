評估「換 vendor」「重構某模型」「導入第三方依賴」「遷移雲商」這類 strategic 決策 — 不能只算直接金流（每月省 $X）。要把 **ops 痛 / 中斷風險 / 合規價值 / vendor lock-in** 一併算進去，否則 ROI 評估是 misleading。

**Why:**
- 直接金流是最容易算的維度，但通常是最小的一塊
- 真正貴的是 ops 持續成本（on-call 痛 / 環境維護 / 升級節奏）+ 風險（中斷成本 / 合規違規 / vendor 倒閉）
- 把這些算進去後，「省 $720/yr」可能反而變「淨損失 $5000/yr」
- 反過來「貴 $200/mo」可能變「淨省 $3000/yr」當算入 ops 痛降低
- 用戶看單一金流數字決策 = 系統性低估「永續維護」這條 — 必須主動 surface 給用戶完整 model

**How to apply:**

評估遷移類決策時，提案中 must 含**完整 ROI 表**（不只直接金流）：

| 軸線 | 換 A 的成本 | 換 B 的成本 | 差距 |
|---|---|---|---|
| **直接金流** — 每月帳單 | $X | $Y | $(X-Y) |
| **Ops 痛** — on-call 頻率 / 環境維護 | N hours/mo | M hours/mo | (N-M) × $rate |
| **中斷風險** — 預期 down time × 每分鐘損失 | D₁ min × $loss | D₂ min × $loss | (D₁-D₂) × $loss |
| **遷移成本** — 一次性人天 + 機會成本 | T₁ 天 × $rate | T₂ 天 × $rate | (T₁-T₂) × $rate |
| **合規** — 對 GDPR / SOC2 / 國別 / 行業特殊規定 | OK / 漏 | OK / 漏 | 合規違規一次 $> 一年總金流 |
| **Vendor lock-in** — 退出成本（半年內 / 5 年內） | 低 / 中 / 高 | 低 / 中 / 高 | 高 lock-in = 應加 30% premium |
| **學習曲線** — 新團隊上手時間 | 1d / 1w / 1mo | 同 | 影響招募 + onboarding |
| **生態** — 文件 / community / hire-ability | 強 / 弱 | 同 | 弱生態長期成本翻倍 |

**Surface 給用戶的格式**：
1. 列直接金流（最容易）
2. **接下來「但要算 ops + 風險 + 合規」一段** — 不要等用戶問
3. 給總和後的「實際年化淨成本」
4. 如果跟直接金流結論不同 — 明確說「直接金流看起來 A 划算 $X/yr，但全成本算下來 B 反而省 $Y/yr 因為 Z」

**範例（generic）**：
> 直接金流：選 A 月省 $50（年 $600）。
>
> 但 A 的 ops 痛：每月需 2 人時 / 開發者額外維護 → $240/mo（市場行情 $120/hr）。
> A 的中斷風險：上次同類 outage 19 分鐘 / $300 損失。預期 / 年 12 次 = $3600/yr。
> A 的學習曲線：3 人 × 2 週上手 → 一次性 $36000，年化分攤 5 年 = $7200/yr。
>
> 全成本年化：A = $-600 + $2880 + $3600 + $7200 = $13080
> B = $0 (baseline) + $0 + $0 + $0 = $0
>
> **B 反而年省 $13080**，雖然每月直接金流貴 $50。建議 B。

**例外（可以單純看直接金流）**：
- $/月差距 $1000+ 而且 ops / 風險 / 合規 都同等 — 那直接看金流 OK
- 短期實驗（< 1 個月）— 沒有「永續」這條軸
- 個人 prototype / 玩具專案 — 沒有 ops 痛
